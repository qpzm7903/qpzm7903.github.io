某团队架构评审会上，一张包含了 15 个微服务的"宇宙全量大图"铺满了屏幕。密密麻麻的方框和交织如麻的箭头让在座的专家们沉默了整整 30 秒——没人能一眼看清核心逻辑，也没人敢轻易断言设计是否合理。

这种场景在技术团队中并不罕见。大多数架构文档被吐槽"看不懂"或"太细碎"，根源不在于绘图者的信息掌握不足，而在于缺乏**分层视角**。

核心判断如下：一份合格的架构方案必须通过四层递进逻辑构建——上下文图定边界、容器图看分组、流分析理交互、概念模型抓核心。只有逐层剥离复杂度，才能让复杂的分布式系统既展现全局"森林"，又刻画局部"树木"。

## 第一节：为什么需要分层视角

在面对超过 10 个服务模块的系统时，试图用一张图解释所有细节是徒劳的。架构图最常见的失败有三类：

1. **信息过载**：把部署节点、代码类名、外部接口全部塞在一起
2. **层次混乱**：在描述业务逻辑的图里，突然出现了一个 Redis 集群
3. **内外不分**：分不清哪些是系统内部组件，哪些是外部依赖

四层法的核心思想是：**每一层只回答一个问题**。通过限制每一层的视角，强迫设计者在不同抽象维度上进行思考，从而确保信息的传达是精准且高效的。

## 第二节：第一层——系统上下文图（回答"谁在圈外"）

系统上下文图（System Context Diagram）是架构设计的起点。它的目标是定义系统的**边界**。

**画法口诀**：一个黑盒 + 外部实体 + 粗粒度交互。

在这一层，我们将整个待建系统视为一个不可分割的"黑盒"。以"智能快递柜系统"为例，它的外部实体包括用户、微信支付平台和柜机硬件。

**常见错误**：在上下文图中画出数据库、消息队列或具体的微服务。在这一维度，这些都属于"盒内"细节，不应出现。

```text
+-------------------------------------------------------------+
|                    系统上下文图 (System Context)              |
+-------------------------------------------------------------+
|                                                             |
|  +-----------+     +----------------+     +--------------+  |
|  |   用户    |     |    微信平台     |     |   柜机硬件    |  |
|  +-----+-----+     +--------+-------+     +------+-------+  |
|        |                   |                    |          |
|        | (扫码取件)        | (支付/通知)        | (状态上报) |
|        +------------------>|                    |          |
|        |                   +------------------->|          |
|        |                                          |          |
|        +------------------+----------------------+          |
|                           |                                 |
|                    +------v-------+                          |
|                    |  智能柜系统   |                          |
|                    |   (黑盒)     |                          |
|                    +--------------+                          |
|                                                             |
+-------------------------------------------------------------+
```

## 第三节：第二层——系统组成图（回答"内部怎么分组"）

打开黑盒后，我们面对的是一群微服务。与其直接罗列 15 个服务，不如先进行**逻辑分组**。

**分组原则**：按"业务职责 + 流处理角色"双维度切分。

1. **用户交互层**：处理 App、H5、小程序等流量入口
2. **业务处理层**：核心领域逻辑，如订单、格口管理、支付中心
3. **外部网关层**：专门负责对接硬件协议或第三方 API

对于基础设施，在这一层只保留**数据库（DB）和消息队列（MQ）**这两大件，用于表示数据落盘和异步解耦。

```text
+-----------------------------------------------------------+
|                   智能柜系统 (内部组成)                    |
+-----------------------------------------------------------+
| [用户交互组]                                              |
|  +------------+      +------------+                      |
|  |  用户 App   |      |  后台管理   |                      |
|  +------------+      +------------+                      |
|        |                    |                             |
|        v                    v                             |
+-----------------------------------------------------------+
| [业务处理组]                                              |
|  +------------+      +------------+      +------------+  |
|  |  订单中心   | <--> |  格口管理   | <--> |  支付中心   |  |
|  +------------+      +------------+      +------------+  |
|        |                    |                    |        |
|        +--------------------+--------------------+        |
|                            |                              |
|                    +-------v-------+                       |
|                    |   DB / MQ     |                       |
|                    +---------------+                       |
+-----------------------------------------------------------+
| [外部网关组]                                              |
|  +------------+      +------------+                      |
|  |  柜机控制台 |      |  微信对接端 |                      |
|  +------------+      +------------+                      |
+-----------------------------------------------------------+
```

## 第四节：第三层——上下行流分析（回答"什么流走哪里"）

有了组成图，系统依然是静止的。流分析层通过定义不同的"流"类型，让架构动起来。

我们定义三种色系（或线型）的流：
- **🔴 同步下行流**：用户发起请求，系统实时响应（如：查询快递）
- **🔵 异步上行流**：硬件或外部系统上报状态，系统异步处理（如：柜门关闭信号）
- **🟢 被动采集流**：定时任务或埋点采集（如：日终对账）

**流归属判断表**：

| 流名称 | 触发源 | 核心路径 | 性质 |
| :--- | :--- | :--- | :--- |
| 扫码取件 | 用户 | 交互组 -> 处理组 -> 外部网关(柜机) | 同步下行 |
| 硬件心跳 | 柜机 | 外部网关 -> 处理组(状态更新) | 异步上行 |
| 支付回调 | 微信 | 外部网关 -> 处理组(订单完成) | 异步上行 |

```text
[用户] --(🔴扫码取件)--> [交互组] --(🔴验证)--> [处理组]
                                            |
                                     (🔴指令) |
                                            v
[柜机] <--(🟢上报状态)-- [外部网关] <---------+
  |                                         ^
  +--------(🔵关门信号)---------------------+
```

通过在容器图基础上叠加这些流，你可以一眼看出哪些分组是"流量黑洞"，哪些分组是"中转枢纽"。

## 第五节：第四层——核心概念与能力地图（回答"到底能干啥"）

前三层解决了"在哪"和"怎么走"的问题，最后一层要深入本质：这个系统承载了哪些**核心模型**和**业务能力**？

### 1. 概念模型
不要画成包含几十个字段的 ER 图。这里只需要画出跨组引用的 5-8 个**聚合根**。
- **Order (订单)**：生命周期的核心
- **Locker (格口)**：物理资源的抽象
- **PickupCode (取件码)**：凭证概念

```text
+----------+      1:1      +----------+
|  Order   |-------------->|PickupCode|
+----------+               +----------+
     | 1
     |
     | n
+----------+      1:1      +----------+
|  Locker  |-------------->|  Box     |
+----------+               +----------+
```

### 2. 能力地图
用一张表补足行为视角。能力不是简单的 CRUD（增删改查），而是具有业务价值的动作。

| 能力域 | 核心能力 | 触发源 | 暴露方式 |
| :--- | :--- | :--- | :--- |
| 投递域 | 远程开门 | 后台管理员 | Dubbo/HTTP |
| 履约域 | 取件码核销 | 用户小程序 | API Gateway |
| 运维域 | 格口故障锁死 | 监控系统 | 内部消息 |

**两层互补**：概念模型提供静态结构，能力地图提供动态行为。两者结合，才是完整的"系统说明书"。

## 第六节：落地 checklist 与常见坑

**交付顺序建议**：
1. **上下文图**：先对齐外部依赖，防止评审中途突然跳出个"没考虑到的第三方系统"
2. **容器/组成图**：确定服务分组，分配团队职责
3. **流分析**：跑通核心链路（Happy Path）
4. **概念/能力**：定义标准语言，确保产品、开发、测试沟通在同一频道

**三大坑**：
1. **流图 vs 部署图**：流图展示逻辑交互，不要在上面标注"这是 Nginx 节点"或"这是北京机房"
2. **概念模型 vs ER 图**：概念模型是逻辑上的关联（Order 包含 PickupCode），ER 图是物理上的表结构
3. **能力地图 vs CRUD 列表**：能力地图应写"包裹入库"，而非"Insert Order Table"

**工具建议**：
不要纠结于高级建模工具。推荐 **draw.io**（或其开源版 Diagrams.net），生成的 XML 文件可以存入 Git 仓库。通过 Confluence 的插件嵌入，并保持版本化的 PNG 输出，确保文档与代码同步更新。

## 本文思维导图

```text
架构设计四层法
├── 第一层：系统上下文图 (Context)
│   ├── 核心目标：定义边界、识别外部实体
│   └── 关注点：黑盒视图、外部依赖、粗粒度交互
├── 第二层：系统组成图 (Container)
│   ├── 核心目标：逻辑分组、明确技术栈
│   └── 关注点：用户交互组、业务处理组、网关组、基础设施
├── 第三层：上下行流分析 (Flow)
│   ├── 核心目标：动态链路、时序梳理
│   └── 关注点：同步下行(红)、异步上行(蓝)、被动采集(绿)
└── 第四层：概念模型与能力地图 (Concept/Capability)
    ├── 核心目标：业务本质、静态与动态结合
    └── 关注点：聚合根(模型)、能力域(行为)、触发源与暴露方式
```

**结语**：
好的架构图应该是自顶向下的"剥洋葱"过程。当你下次准备画图时，先问问自己：我这一层到底是在回答哪个问题？如果一张图承载了超过两个层级的答案，那就请果断地把它拆开。
