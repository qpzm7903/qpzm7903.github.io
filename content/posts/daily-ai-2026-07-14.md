## ⭐ 今日最值得关注

### 1. 同一份 TypeScript 在 Claude 上比 GPT 多花 73% token——有人精确统计了每个前沿模型的 tokenizer

playcode.io 团队做了一件简单但没人做过的事：拿 16 份真实内容（英文散文、HTML、JavaScript、Python、TypeScript、Rust、JSON tool schema、中文聊天、一个真实的 agent system prompt），在每家厂商的**官方 tokenize 接口**上逐字节统计 token 数，再乘以公开标价算出"有效价格"。HN 123 分。核心发现：

- **Anthropic 新 tokenizer 是一次"悄无声息的涨价"**：Claude Sonnet 5 / Opus 4.8 / Fable 5 的新 tokenizer 对同一份 TypeScript 产生 1,178 tokens，旧 tokenizer 只产生 898 tokens——**+31%**，且标价不变。英文 system prompt 场景涨幅 +39%。中文几乎不变（+0%），说明膨胀集中在英文和代码。
- **跨厂商对比，代码品类差距最大**：以 GPT 的 `o200k` tokenizer 为基准（1.00x），Claude 新 tokenizer 对 TypeScript 为 **1.73x**、Rust 1.58x、JavaScript 1.52x、Python 1.50x——而英文散文只有 1.40x。GPT tokenizer 对 TypeScript 达到 4.24 字符/token（明显在 web JS/TS 上调优过），Claude 则在所有代码语言上差不多密度。
- **Sonnet 5 的"降价"是过渡缓冲不是真降价**：intro 价格 $2/$10 到 8 月 31 日结束，之后恢复 $3/$15。加上 +32% token 膨胀，9 月 1 日起同一份代码在 Sonnet 5 上比 Sonnet 4.6 多花约 32%，而标价完全一样。
- **用真实 API 账单验证**：团队发了 `max_tokens:1` 的真实付费请求，Opus 4.6 billed 2,541 input tokens，Opus 4.8 billed 3,191——与 `count_tokens` 预测逐一吻合，膨胀体现在账单上不是估算。

对 coding agent 用户和做成本工程的团队，这篇给出了一种可复制的比较方法：**不要只看 $/Mtok 标价，要看"你的真实 payload 在这个 tokenizer 下变成多少 token"**。

> 「原文金句」: "Nobody prints the tokens-per-content number, so nobody compares it. So we measured it."

"没有任何厂商公布'每份内容的 token 数'这个数字，所以没人去比较它。于是我们去量了。"

- 原文：[The Same TypeScript Costs 73% More on Claude Than on GPT](https://playcode.io/blog/real-price-of-frontier-models)

### 2. Apple SpeechAnalyzer 首次精确基准：在 LibriSpeech 上超越 Whisper Small，速度快 3 倍

inscribe 团队因为自己的产品同时内置了 Apple 两个引擎和三个 Whisper 模型（WhisperKit CoreML），得以在同一个 M2 Pro、同一套生产代码路径上跑 LibriSpeech 完整 clean + other 共 5,559 条语音。HN 380 分。结果：

- **SpeechAnalyzer（iOS/macOS 26 新 API）全面领先**：test-clean WER 2.12%，test-other WER 4.56%——低于 Whisper Small（3.74% / 7.95%），低于 Whisper Base（5.42% / 12.51%），也大幅低于旧 SFSpeechRecognizer（9.02% / 16.25%）。旧 API 到新 API 的 WER 降幅达 3.5~4 倍。
- **速度也是 3 倍**：SpeechAnalyzer 在比 Whisper Small 更准确的同时，每秒音频的推理时间约为 Whisper Small 的 1/3。所有引擎在 M2 Pro 上均跑 12x~40x 实时速度。
- **验证方法值得学**：团队用 LibriSpeech 恰是因为 OpenAI 公开了 Whisper 在该数据集上的 WER——他们的 Whisper 测量值与 OpenAI 公开值的偏差在 +0.11~+0.42 个百分点之间且全为正偏（一致的 text normalizer + CoreML 量化导致），不是随机散布，说明测量管道可信。
- **产品结论**：inscribe 已将 Auto 引擎默认改为优先 SpeechAnalyzer（支持约 30 种语言），Whisper 用于其余语言。Whisper 仍有跨平台和多语言优势，但"英文 Whisper 是 Apple 硬件上自动首选的时代结束了"。

> 「原文金句」: "Shipping a benchmark and ignoring it in your own defaults would be a strange kind of honesty."

"发布一个基准测试却在自己产品的默认设置里无视它，那将是一种奇怪的诚实。"

- 原文：[Apple's New Speech API vs Whisper: The First Real Benchmark](https://get-inscribe.com/blog/apple-speech-api-benchmark.html)

### 3. 不开 Xcode 就能 build 和 ship Mac/iOS app——一个 vibe coding 完整工作流

Scott Willsey 发了一篇实操指南，展示了完全用命令行 + AI coding agent 构建、签名、公证和部署 Mac/iOS 应用的完整路径。HN 194 分。对正在用 Claude Code / coding agent 的人是直接可用的工程参考：

- **关键前提**：`xcodebuild`、`notarytool`、`stapler`、`devicectl` 都住在 Xcode.app 里，但在 shell 里跑就够了，不用打开 Xcode GUI。
- **一次性 GUI 设置**：签入 Apple ID、创建 Developer ID Application 证书、存储公证密码——之后全 headless。
- **XcodeGen 管理项目文件**：用 `project.yml` 生成 `.xcodeproj`，只有 YAML 需提交 git，不再有 `.xcodeproj` 持续修改导致的 git 冲突。
- **一条脚本完成发布链**：`scripts/release.sh` 跑完 archive → Developer ID 签名 → 票据公证 → staple → 安装到 `/Applications`。签名密钥在 login keychain 里，`xcodebuild` 自动找到。
- 他明确提到可以"把这篇博客喂给 Claude Code 或你的 LLM coding 工具让它自己弄"——把文档作为 agent 上下文的一部分是真实可用模式。

原文：[Building and Shipping Mac and iOS Apps Without Ever Opening Xcode](https://scottwillsey.com/building-and-shipping-mac-and-ios-apps-without-ever-opening-xcode/)

## 📖 今日英语

**tokenizer** — "Each model's tokenizer cuts the same file into a different number of pieces, and you pay per piece."（出处：playcode.io 前沿模型定价文章）
"分词器"——将文本切分为计费单元的组件。同一份文件在不同 tokenizer 下产生的 token 数不同，直接决定你实际支付多少钱。今天文章的核心观点就是：大家只看 $/Mtok 标价却忽视了"我的内容在对方的 tokenizer 下变成多少 token"。常见搭配：tokenizer efficiency, train a tokenizer, o200k tokenizer。

**sticker price / list price** — "Same vendor, same sticker price, ~30% more tokens."（出处：playcode.io 文章）
"挂牌价 / 公开标价"——在讨论 API 定价时指厂商在定价页对外公布的数字，和"有效价格"（effective price）相对。playcode.io 的关键洞察是：sticker price 不变但 tokenizer 膨胀 = 隐形涨价。常见搭配：at sticker price, below sticker price, intro price vs list price。

**WER (Word Error Rate)** — "SpeechAnalyzer 2.12% / Whisper Small 3.74% / SFSpeechRecognizer 9.02%"（出处：inscribe Apple Speech API 基准文章）
"词错率"——语音识别的标准评测指标，指引擎在转写时替换、丢弃或编造的词占比。WER 越低越好。LibriSpeech test-clean（2,620 条干净朗读语音）和 test-other（2,939 条更嘈杂语音）是通用基准。常见搭配：clean/noisy WER, per-utterance WER, corpus WER (not averaged WER)。

**headless** — "After that, builds and deploys are fully headless."（出处：scottwillsey Xcode-free 工作流文章）
"无界面 / 命令行执行"——指不需要打开 GUI 应用就能完成任务。与"GUI-guided"相对。在 agent 时代，headless 工作流是 agent 可操作的前提：agent 不能"点击 Xcode 窗口"，但能跑 `xcodebuild` shell 命令。常见搭配：headless build, headless browser, headless deploy。

**consent toggle** — "Samsung has introduced a 'Consent to the Use of Health Data for AI training and modelling' toggle deep in the settings."（出处：Samsung Health / Neowin 报道）
"同意开关 / 授权切换选项"——指 app 设置中让用户同意或拒绝数据被用于 AI 训练的二选一控件。Samsung Health 的做法（关闭即删除数据）引发了"到底是 consent 还是 coercion"的争论。常见搭配：AI data consent toggle, opt in / opt out toggle, GDPR consent。
