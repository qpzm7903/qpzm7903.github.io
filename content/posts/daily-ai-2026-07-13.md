## ⭐ 今日最值得关注

### 1. 陶哲轩实测 AI 编码 agent：24 个旧 applet 几小时迁移成功，新想法 26 年后终于成真

陶哲轩（Terence Tao）在博客分享了一次用 AI 编码 agent 的完整体验，HN 上以 387 分 / 111 评论成为周末最热帖。他做了三件事：

1. **迁移旧代码**：把自己 1999 年用 Java 1.0 写的数十个数学可视化 applet（蜂蜜梳图、Besicovitch 集、复分析课件等）交给 agent 端口到现代 JavaScript，几小时内全部恢复可运行，还自动上了色。整批迁移只发现一个 math.abs 处理拖拽事件的小 bug——同时 agent 反过来发现了他原代码里的两个隐藏 bug。
2. **实现 26 年前的旧想法**：1999 年他设想了一个"在 Minkowski 空间里的 Inkscape"——一个用于可视化狭义相对论的时空图工具，但当时手写 Java 太复杂放弃了。经过几小时"vibe coding"后，这个 26 年前的设想终于实现了，[现在可以在线使用](https://teorth.github.io/tao-web/apps/spacetime-diagram.html)。
3. **为新论文做交互可视化**：当天写完 Gilbreath 猜想的博文后，几个小时内又让 agent 做出了[配套可视化工具](https://teorth.github.io/tao-web/apps/gilbreath.html)。

对一个拿过菲尔兹奖的数学家来说，核心信号非常具体：**LLM agent 产出的代码仍然有 bug，但 bug 的数量和重要性已经低到可以用"不是关键路径"的 supplementation 形式安全使用**。他说自己可能会把这类交互式可视化作为未来论文的标准附件——在非 mission-critical 的场景下，agent 的风险/收益已经划算。

- 原文：[Old and new apps, via modern coding agents](https://terrytao.wordpress.com/2026/07/11/old-and-new-apps-via-modern-coding-agents/)

### 2. Claude Code 每次发请求前要塞进 33k tokens，OpenCode 只要 7k——有人精确测了

Systima.ai 团队做了一次少见的"精准称重"实验：在 Claude Code 2.1.207 和 OpenCode 1.17.18 之间插入一个 logging proxy，让两者在同一个模型（Claude Sonnet 4.5）、同一个任务下运行，记录每个请求的精确字节数和 API usage。HN 353 分 / 199 评论的核心发现：

- **"说一声 OK"就要花 33k tokens**：Claude Code 在首次请求里塞了 27,344 字符的系统 prompt（3 个 block）、99,778 字符的 27 个工具 schema、7,997 字符的 `<system-reminder>` 脚手架；OpenCode 只有 9,324 字符系统 prompt（1 个 block）和 20,856 字符的 10 个工具 schema。
- **cache 效率差 54 倍**：OpenCode 的请求 prefix 每字节位完全一致，一次 cache 写、后续按缓存读便宜；Claude Code 在会话过程中反复重写数万 prompt-cache token，同一个任务最多比 OpenCode 多写 54 倍 cache token——cache 写按溢价计费，所以仪表盘涨得飞快。
- **配置会继续膨胀**：一个真实仓库的 CLAUDE.md（72KB）增 20k token，5 个 MCP server 再加 5~7k。真实生产环境里用户还没开始打字，请求就堆到了 75k~85k tokens。
- **subagent 的"扇出成本"是 4.2 倍**：一个 121k token 的小任务拆给 2 个 subagent 要花 513k tokens——每个 subagent 都付一遍 bootstrap，父 agent 还要消化它们的 transcript。
- **Claude Code 也有一处胜出**：多步任务里它会把工具调用打包到更少的请求里，OpenCode 每轮要重付较小的 baseline，所以谁最终更贵取决于会话如何展开。

对生产环境部署 agentic AI 的团队，这篇给出了一组有数据支撑的 cost engineering 指标——尤其是在 EU AI Act Article 12 要求"能用数据回答你的 agent 到底发了什么"的前提下，这是一份很扎实的参考样本。

- 原文：[Claude Code Is Way More Token-Hungry Than OpenCode. We Measured Exactly How Much](https://systima.ai/blog/claude-code-vs-opencode-token-overhead)

### 3. geohot 二度发文："我爱 LLM，我讨厌 hype"——自己用 opencode+GLM-5.2 装 tmux 都成功了

[comma.ai](https://comma.ai/) / [tinygrad](https://tinygrad.org/) 创始人 George Hotz（geohot）继 7 月 11 日那篇《AI 2040 and the cult of intelligence》之后，于 7 月 12 日再发短文《I love LLMs, I hate hype》，HN 256 分 / 139 评论——这次不是反驳，而是自我定位。他说自己从 2007 年到现在整个 career 都在干 AI——"从这个博客你可能会误解我对 AI 有多 giddy"。具体内容：

- **他自己上周装了一台 Linux + opencode + 本地 GLM-5.2 的机器，然后用 `install tmux with the geohot configuration` 这种自然语言指令真的搞定了**：原话是"the Year of the Linux Desktop is finally here!"。
- **他不喜欢的只有两件事**：第一是"窗口正在关闭/永远落后"这类 negative valence 的 hype——指出这东西主要目的是让你对自己感觉不好、然后搬到他想骂的旧金山去；第二是从"fancy autocomplete"直接跳到"眨眼间吞噬整条光锥"的稻草人 extrapolation——他赌上身家说这不会发生。
- 关于编程生产力：他引用了 Linus Torvalds 的说法——compilers 让编程 1000 倍效率、agents 让 10 倍——他承认自己之前在《The Eternal Sloptember》里对模型写代码能力的评价太苛刻了，编程本身正在变化，agent 对他个人有可测的助益。但他也明确说：**所有 vibe coded 的东西仍然是 slop**（"where's all this new magical software that the productivity improvements should imply?"），模型可以用但要小心，因为它会"增加 cognitive fatigue"。
- 他自述的判断：**"AI 是计算机革命的延续。我爱计算机所以爱 AI。"** ——把 AI 放回"通用计算进步"的延长线，而不是"奇异 godlike entity"那条线。

跟 7 月 11 日的文章放在一起读，两篇加起来是当下对 AI 叙事最有代表性的一组"反方"声音：不是反对 AI 本身，而是反对它被某些"实验室 + SF 派系"占为己有的过程。对工程师读者，第二篇最有用的部分是那个本地 opencode + GLM-5.2 的实操经验——一个公开对抗前沿实验室估值叙事的创始人公开换到了"开源 + 本地"的工作流。

- 原文：[I love LLMs, I hate hype](https://geohot.github.io/blog/jekyll/update/2026/07/12/i-love-llms.html)

## 📖 今日英语

**giddy** — "I think from this blog you may misunderestimate how absolutely giddy I am about AI."（出处：geohot「I love LLMs」原文）
"雀跃 / 兴奋得发晕"，形容词，比 excited 更"心跳加速"的语义。geohot 用这个词刻意把自己对 AI 的真实热情摆出来，以免读者把他前一篇对 hype 的反驳误读为"对 AI 本身失望"。常见搭配：giddy about/in love with、giddy excitement。

**harness** — "We put Claude Code and OpenCode on the same model, the same machine, and the same tasks..." / "harness (Claude Code / OpenCode) → logging proxy → model endpoint"（出处：Systima token overhead 文章）
"(agent) 框架 / 套壳"——把模型包装起来、给它工具、循环推理的软件层就叫 harness。Systima 的文章里反复用这个词指代 Claude Code 和 OpenCode 本身。常见搭配：agentic harness、test harness、CI harness——这是一个不分前端/后端/Agent/测试工程师都会遇到的词。

**cache write / cache read** — "Cache writes of course are billed at a premium, which accounted for the usage dashboard climbing when using Claude Code."（出处：Systima token overhead 文章原文）
"缓存写 / 缓存读"——LLM API 的 prompt caching 计费模型里，首次写入长 prompt 按更高的"cache write"价格计，后续命中按极低的"cache read"计。理解这条区别对成本优化的工程师是基础——如果 harness 每轮都重写 prompt，cache write 会反复发生、账单就涨得快。常见搭配：cache write premium、cache hit/miss、prompt cache。

**slop** — "But models are useful just like find replace, stack overflow, or all the regexes I never learned how to write and now never will! ... AI is the continuation of the computer revolution, I love computers so much. ... all the vibe coded stuff is still slop"（出处：geohot 原文）
"劣质品 / 敷衍货"，2024 年开始在 AI 圈成为专有词：特指 AI 生成出来但未经认真打磨、没真正解决问题的大量内容。geohot 对 vibe coding 的态度保留——"有用但仍然是 slop"——是一个重要且诚实的判断。常见搭配：AI slop、vibe-coded slop、slop content。

**raw** — "regularMarketPrice": {"raw": 7575.39, "fmt": "7,575.39"}（出处：Yahoo Finance API 返回结构）
Yahoo Finance 在 API 返回中给每个价格字段同时给两个值：`raw` 是机器可读的数字，`fmt` 是带千位分隔 comma 的字符串。要拿来做计算必须读 `raw` 而不是 `fmt`——这是所有直接调用财经 API 的工程师都会遇到的小坑。常见搭配：raw value、raw number、raw field vs formatted field。
